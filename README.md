# Aegis — an AI meal planner with a safety shield

**Live:** https://aegis-zeta-six.vercel.app · **Repo:** https://github.com/yarrbakr/Aegis

Aegis generates a personalized **weekly meal plan** from a user's diet, budget, household size, allergies and taste — and wraps the AI in a **deterministic safety guardrail** that screens every generated meal against the user's declared allergens *before it is ever shown*. Unsafe suggestions are blocked, regenerated, and logged. The safety claim isn't a promise — it's a number you can reproduce with one command.

> The headline isn't "an AI meal planner." Everyone can build that. The headline is:
> **"How do you know the AI is right, and what happens when it's wrong or attacked?"** — answered, visibly, on screen.

Built for the CyberGen / Himmatkaar AI-First Developer task. The whole thing was built with **Claude Code**, and the process is a graded artifact — see [the decision log](playbook/Memory.md), [the prompt log](playbook/Prompt.md), and [the build documentary](Documentary.md).

## The one idea that matters

**The LLM suggests. It never approves safety.** Every meal the model produces is checked by **deterministic TypeScript** — not an "is this safe?" prompt to another model. A security company never lets the model grade its own safety. That principle holds *everywhere* in Aegis: planning, single-meal regeneration, snack recommendations, and (deliberately) it would hold in the deferred chatbot too.

There are two guardrails:

- **Output guardrail** (`lib/guardrails/allergen.ts`) — `screenMeal()` scans each meal with **defense in depth**: not just the model's `allergen_tags`, but ingredient names and the meal text too, synonym-expanded, and allergen-aware so a "dairy-free cheese" or "gluten-free bread" doesn't nuisance-trip. Unsafe → block, log, regenerate (bounded); if still unsafe, a guaranteed-safe placeholder — **never the unsafe meal.**
- **Input guardrail** (`lib/guardrails/injection.ts`) — the user's free-text preferences (allergens *and* taste fields) are screened for prompt-injection (`"ignore previous instructions…"`), the offending text is dropped before it reaches the model, and the attempt is logged. The system prompt is fixed server-side; **user input is inserted as data, never as instructions.**

Every decision is written to a `safety_events` audit log that powers a live **Security Console** in the app.

## Proof, not vibes — the eval

`npm run eval` builds a **labeled** corpus of unsafe and safe meals across 14 allergy profiles and runs every item through the **real shipped** `screenMeal()` (not a copy). Unsafe meals hide a real allergen three ways — correctly tagged, an **untagged** ingredient, and **meal-name-only** — so it measures real coverage, not the guardrail grading itself.

```
236 meals (180 unsafe / 56 safe) across 14 profiles
→ CATCH RATE   180/180 = 100.0%   (every unsafe meal blocked)
→ SPECIFICITY   56/56  = 100.0%   (no safe meal wrongly blocked)
```

The process exits non-zero on a single miss. `npm run test:guardrail` adds 27 focused unit checks (untagged catches, "free-of" false-positive guards, injection screening, taste matching). Full output: [`lib/eval/RESULTS.md`](lib/eval/RESULTS.md).

## Features

- **Auth & onboarding** — email/password (Supabase Auth); diet, allergies, budget, household, and **taste preferences** (favorite cuisines + foods to skip), saved per user under Row-Level Security.
- **Weekly plan generation** — 7 days × breakfast/lunch/dinner with ingredients, cost (USD) and macros, generated by Groq, Zod-validated, then guardrail-screened before display.
- **Taste, done honestly** — favorite cuisines and dislikes shape the plan, but they're **best-effort** and clearly labelled as such: only declared **allergens** are guardrail-enforced. Dislikes are *also* enforced deterministically (a meal containing a disliked food is re-rolled before the safety pass) — but if a dislike ever conflicts with a safe, on-diet meal, **safety wins.**
- **Dashboard** — pastel stat cards, a weekly calories chart, a macro donut, a "meal for today" panel, and **snack recommendations that pass the same guardrail** as the plan.
- **Security Console** — the `safety_events` audit trail, humanized into "here's what we caught for you" cards: safety made legible, not scary.
- **Resilience** — Groq primary, **Mistral fallback** if Groq is down or rate-limited; a provider rate-limit surfaces as a friendly "at capacity" message, not a crash.

## Stack (and why)

| Layer | Choice | Why |
|---|---|---|
| Frontend | **Next.js 16** (App Router, TypeScript) | Vercel-native; the "boring on purpose" pick. |
| Styling | **Tailwind CSS v4** + shadcn-style components | Fast, tasteful, accessible. |
| Charts / motion | **Recharts**, **Motion** | Budget/nutrition/calorie charts; landing hero animation. |
| Database | **Supabase (Postgres)** | The data is relational (user → plans → meals → ingredients). |
| Auth + access | **Supabase Auth + Row-Level Security** | Real per-user isolation, not decoration. |
| AI + guardrail | **Next.js Route Handler** (server, on Vercel) | One platform; runs under the user's session so **RLS is enforced** on every write. |
| LLM | **Groq** `llama-3.3-70b-versatile` (primary) · **Mistral** `mistral-small-latest` (fallback) | Fast + free; a second provider = resilience. |
| Validation | **Zod** | Never trust unvalidated data — especially LLM output. |
| Deploy | **Vercel** + Supabase hosted | A working link beats a perfect stack. |

`/backend` holds a **FastAPI** mirror of the architecture (documented; a post-core Render "evidence endpoint" stretch). The primary, shipped path is all-Vercel — see decision **D10**.

## Data model

```
profiles          id → auth.users · display_name · diet_type · allergens[]
                  · favorite_cuisines[] · disliked_foods[] · weekly_budget · num_people
meal_plans        id · user_id → profiles · week_start · total_cost · status
meals             id · plan_id → meal_plans · day_of_week · meal_type · name · description
                  · cost · calories · protein_g/carbs_g/fat_g · safety_status
ingredients       id · meal_id → meals · name · quantity · allergen_tags[]   ← the guardrail scans these
safety_events     id · user_id · plan_id · event_type · allergen · detail    ← audit log → Security Console + eval
```

Relational on purpose (`profiles 1→∞ meal_plans 1→∞ meals 1→∞ ingredients`). **RLS on every table** — a user can only ever touch their own rows. DDL in [`supabase/schema.sql`](supabase/schema.sql) + [`supabase/policies.sql`](supabase/policies.sql); the taste columns in [`supabase/migrations/0001_taste_preferences.sql`](supabase/migrations/0001_taste_preferences.sql).

## Run it locally

```bash
npm install
npm run dev            # http://localhost:3000

npm run test:guardrail # 27 focused guardrail unit checks
npm run eval           # the labeled catch-rate harness (→ 100% / 100%)
npm run build          # matches the Vercel build
```

Environment (`.env.local`, all gitignored — see [`.env.example`](.env.example)):

```
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
GROQ_API_KEY=...            # server-side only
MISTRAL_API_KEY=...         # optional — enables the fallback
```

> **Note:** the LLM keys are **server-side secrets** — never `NEXT_PUBLIC_`, never committed. Groq's free tier has a daily token cap; setting `MISTRAL_API_KEY` gives the app a fallback when that cap is hit.

## Boundaries (what Aegis is *not*)

- **Not a medical device.** It filters *declared* allergens; it does not give medical or nutritional advice, and the UI says so.
- **No real payment/bank integrations.** Out of scope by design.
- Dislikes are **taste hints**, not a safety guarantee — only allergens are enforced.

## What I'd do next (kept honest)

Naming the limits is itself the point — evidence over hype.

- **Conversational assistant** — a Mistral-based chat widget (kept off Groq's quota) scoped to *help*, that **defers every safety question to the deterministic guardrail** — never adjudicating safety itself. Designed, intentionally deferred to protect the submission timeline.
- **Groq quota** — the free tier's 100k tokens/day is easily exhausted under heavy testing; a paid tier (or the now-wired Mistral fallback) removes that ceiling. The fallback uses `mistral-small-latest` specifically to stay under Vercel's 60s function limit.
- **Richer allergen intelligence** — the synonym map is hand-curated; a broader ontology (and per-region ingredient names) would widen coverage.
- **RAG over a curated recipe corpus** — ground generation in real recipes rather than pure synthesis.
- **Shopping list + single-meal "swap" button**, and a full dark theme.
- **Deploy the `/backend` FastAPI to Render** as a live Python "evidence" endpoint (matches CyberGen's FastAPI+LLM stack).
